## Core Philosophy

### Context First
Nothing exists in isolation. Every action, post, message, or identity exists within a **context** (Global, Space, Channel).

### Identity is Fluid
A user is not a single static profile. Identity **adapts** per Space, per Channel, per interaction.

### Visuals Communicate Meaning
Colors, borders, glow, motion, and depth are **semantic**, not decorative. They communicate hierarchy, ownership, relevance, and status.

### Modular by Design
Everything in Glow is modular: Spaces, Channels, Entries, Blocks, Themes, Effects. Nothing is hard-coded to one use case.

## Theme System & Cascading Styles

Glow uses a **hierarchical cascade system** for styling.

### Style Priority Order

1. **Object-specific** (entry, block, bubble)
2. **User theme** (if premium)
3. **Channel theme**
4. **Space theme**
5. **Global default theme**

Lower levels override higher levels.

### Synchronization

All visual motion:
- Follows a **global clock**
- Stays **rhythmically consistent**
- Avoids **chaotic animation**

This creates a **coherent, living interface**.


### Architecture

**Clean Architecture** with strict layer separation:

```
Presentation (UI, Widgets, State)
    â†“
Domain (Entities, UseCases, Business Logic)
    â†“
Data (Repositories, DTOs, Mappers)
    â†“
Infrastructure (Supabase, Isar, APIs)
```

**Dependencies flow inward only.** Presentation can depend on Domain, but Domain cannot depend on Data or Presentation.

### Packages

- `glow_core` - Base contracts, utilities, error handling
- `glow_domain` - Entities, use cases, business rules
- `glow_data` - Repositories, data sources, offline-first storage
- `glow_api` - HTTP client, API DTOs, endpoints
- `glow_auth` - Session and authentication
- `glow_realtime` - WebSocket connections, presence, sync
- `glow_blocks_engine` - Entry content engine (Notion-like blocks)
- `glow_blocks_flutter` - Block rendering widgets
- `glow_observability` - Logging, analytics, crash reporting
- `glow_ui` - Design system (atoms, molecules, organisms)
